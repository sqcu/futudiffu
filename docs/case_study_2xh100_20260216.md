# Case Study: First 2xH100 Training Session
## 2026-02-16, PrimeIntellect spot instance

### Summary

50 iterations of REINFORCE policy optimization + 30 macrobatches of BTRM reward
model training ran to completion on 2xH100 (SM90) in 28.7 minutes wall clock.
This was the fourth server restart of the session — the first three were killed
by software defects. The session surfaced 23 defects across 6 categories,
of which 12 were fixed live and the rest documented for follow-up.

The total cost of defects: roughly 90 minutes of wallclock on rented GPUs doing
nothing useful. The total cost of prior engineering: roughly 90 minutes of
wallclock saved by things that worked on the first try.

---

## Part 1: What Broke

### 1.1 Hardcoded paths and platform assumptions

**The n==1 most predictable class of failure.**

`train.py` line 35:
```python
sys.path.insert(0, r"F:\dox\repos\ai\futudiffu\src")
```

`generate_btrm_dataset.py` default args:
```python
parser.add_argument("--output-dir", default=r"F:\dox\repos\ai\futudiffu\btrm_dataset")
```

These are Windows paths. The H100 runs Linux. The fix is `Path(__file__).resolve()`.
This is not interesting as a bug — it's interesting as a *pattern*: local development
produces code that works by coincidence of the developer's environment. The paths
are never wrong locally, so there is no feedback signal until deployment. Python
does not type-check path portability. The developer (human or AI) who writes a
hardcoded path does so because it is the fastest way to make the current line work.
The cost is paid later, by a different context, on a different machine.

**Time cost**: ~5 minutes (catch, fix, rsync, restart). Low because the error is
immediate and obvious. But this class of failure includes `.supersekrit` having a
comment line (`# what is this...`) that `read_text().strip()` grabbed as the HF
token — a silent failure that caused HF uploads to 401 until someone read the
server response carefully. And `~/.ssh/primeintellect_ed25519` having 0644
permissions because the file was copied, not generated — SSH refuses the key
silently with a generic "Permission denied."

**Lesson**: Portability is not tested by running on one machine. Every path, every
file read, every permission assumption is a deployment-time landmine if it was
only ever tested in the environment that created it.

### 1.2 The n==1 fast path: locally sensible, project-scope catastrophic

`LoRALinear.forward()` had an optimization:
```python
if n == 1:
    # Fast path: skip Triton kernel, use plain matmul
    return base_out + (x @ A.T @ B.T) * scale
```

This made sense when it was written. The Triton multi-LoRA kernel has launch
overhead. For a single adapter, two matmuls are cheaper than a kernel launch +
the Triton dispatch machinery. Benchmarks confirmed it: the fast path was faster
for N=1 in isolation.

**The project-scope problem**: `torch.compile` sees those matmuls. Each of the 96
LoRALinear modules with a single adapter (ptheta-only layers) contributed 2 visible
matmul nodes to the compiled graph. 192 matmul nodes triggered full inductor GEMM
analysis — tiling decisions, memory layout, fusion candidates — for each one.
Inductor's compile time is superlinear in the number of visible GEMM operations.

Meanwhile, `multi_lora_op` (the Triton custom_op) is registered with
`torch.library.custom_op`, making it **opaque** to inductor. Inductor sees one
node, emits one call, moves on. Zero GEMM analysis.

**The result**: Compile time was 14+ minutes. Both H100s sat at 0% GPU utilization
while 64 CPU threads churned through inductor passes. After removing the fast path
and routing all N>=1 through the opaque custom_op: 42.5 seconds. A 20x improvement
from deleting an optimization.

**Lesson**: Local optimization and whole-graph optimization are different objectives.
An optimization that makes one operation faster can make the system slower if it
exposes internal structure to a compiler that will obsessively analyze it. The
Triton kernel's opacity is not a limitation — it is a *feature* of the compilation
boundary. This is the same principle as why databases use opaque UDFs to prevent
the query planner from doing something clever-but-catastrophic with user logic.

### 1.3 ZMQ socket races: concurrency bugs that only manifest at N>1

`MultiGPUClient` dispatches work to N inference servers via `ThreadPoolExecutor`.
ZMQ REQ sockets are stateful (REQ/REP ping-pong) and **not thread-safe**. Two
bugs, discovered in sequence:

**Bug 1 (run 2)**: `_broadcast_return_primary()` submitted tasks to all N clients
but only waited for client[0]'s future. If the next broadcast started before
client[1]'s future completed, the pool assigned client[1]'s new task to a
different worker thread → concurrent send() on the same REQ socket → socket
corruption → Server 1 never received `allocate_adapter("ptheta")` → Server 1
compiled with 6 adapters, Server 0 with 108 → asymmetric warmup → stuck.

**Bug 2 (run 3)**: After fixing broadcasts, `generate_trajectories()` round-robin
dispatched 4 jobs across 2 servers: jobs 0,2 → server 0, jobs 1,3 → server 1.
Jobs 0 and 2 both submitted to the pool for the same client. Pool ran them
concurrently → EFSM error ("Operation cannot be accomplished in current state")
→ crash on job 2.

**Fix (run 4)**: Per-client `threading.Lock` instances. Every pool-dispatched
method acquires the target client's lock before touching its socket. Jobs for
different servers run in parallel; jobs for the same server serialize.

**Time cost**: ~30 minutes total across two restarts. The first manifestation
(asymmetric adapter count) was diagnosed from GPU VRAM discrepancy in
`nvidia-smi` (10,735 MiB vs 9,633 MiB). The second (EFSM) was diagnosed from
the error message. Both were fixable in <5 minutes once understood, but
understanding required reading ZMQ protocol state machine documentation.

**Lesson**: `ThreadPoolExecutor` has no thread affinity. A task submitted for
"client X" can run on any worker thread. If you have shared mutable state
per-logical-resource (ZMQ sockets), you need per-resource locks, not per-thread
isolation. This is a standard concurrency pattern but ZMQ's stateful protocol
makes the failure mode particularly sharp: not corruption-over-time but
immediate protocol violation.

### 1.4 inject_lora conflates graph mutation and weight initialization

The `inject_lora` RPC does two fundamentally different things:
1. **Allocate memory, mutate module graph** (add LoRALinear wrappers, register
   buffers, change forward() dispatch) — invalidates torch.compile
2. **Initialize weights** (A/B init, set scale/alpha) — cheap, graph-invariant

Phase 1 compiles with rtheta (6 adapters). Phase 2 calls `inject_lora` for ptheta
to initialize its weights. But `inject_lora` also mutates the graph → full
recompilation → 14+ minutes of idle H100s.

**Fix**: Split into `allocate_adapter()` (graph-mutating, before compile) and
`init_adapter_weights()` (graph-invariant, anytime). Pre-allocate all adapters
at startup with `scale=0.0`, compile once, then toggle scales via
`set_adapter_config()` (registered buffer, invisible to dynamo guards).

**Lesson**: API design that conflates operations with different cost profiles
creates invisible coupling. The caller who needs "initialize weights" gets
"mutate graph + invalidate compile cache" as an undocumented side effect.
The fix isn't more documentation — it's separate entry points with separate
cost models.

### 1.5 Policy rollouts: generate, score, discard

`phase_policy()` generates K rollout trajectories per iteration, scores them with
BTRM, computes REINFORCE advantages, accumulates gradients, then `del trajectory`.
The latent tensors — which are on-policy samples from the current policy
distribution — exist for ~8 seconds and are destroyed.

These are the highest-value samples for future BTRM training: they cover the
distribution the policy actually visits, not the off-policy distribution from
which the original BTRM training set was generated. Every policy iteration
produces 4 trajectories (~12MB) that are scored, used for one gradient step,
and thrown away. Over 50 iterations: 200 trajectories, ~600MB, irrecoverably
lost.

No other code path generates on-policy samples. `generate_btrm_dataset.py`
generates off-policy trajectories (baseline model, no adapters). The persistence
infrastructure (DatasetV2Writer, HF upload) exists and works — it just wasn't
wired into the one code path that produces the most valuable data.

**Fix**: Added `--persist-fraction` / `--persist-min-k` to `phase_policy()`.
Default: persist 1 trajectory per iteration (the highest-reward one) to a v2
dataset in `{output_dir}/policy_rollouts/`. Same DatasetV2Writer, same format,
same HF upload path. 150MB total for 50 iterations. Negligible cost for
irreplaceable data.

**Lesson**: "We can always regenerate it" is only true if the compute environment
still exists. On a spot instance, everything is ephemeral. Persistence should be
the default for any generated artifact that costs GPU-seconds to produce. The
question is not "should we save this?" but "what is the minimum persistence
policy that captures the data we'd regret losing?"

### Recompile spike pattern (run 4, first 6 policy iterations)

```
iter 0: rollout=51.5s  (first compiled forward, Triton kernel JIT)
iter 1: rollout=11.5s  (new prompt length → guard fail → recompile)
iter 2: rollout=69.7s  (new prompt length bucket)
iter 3: rollout=28.7s  (new prompt length bucket)
iter 4: rollout= 7.9s  (cache hit — this prompt length seen before)
iter 5: rollout=36.8s  (new prompt length bucket)
iter 6+: rollout= 7.9s (all prompt length buckets cached)
```

Each unique `cap_padded_len` (prompt length rounded to multiple of 32) triggers a
torch.compile guard failure → full recompile of the forward graph. With 24 prompts
bucketing into ~5-6 unique lengths, the first ~6 iterations eat recompile spikes.
After all buckets are cached, steady-state is 7.9s/rollout.

**Future fix**: FlexAttention shape normalization — use `forward_packed(N=1)` with
a fixed `reference_total_len` (max prompt tokens + image tokens, padded to
reference size) and block_mask to mask padding. All inputs become the same shape
→ single compiled kernel → zero recompile spikes. Infrastructure exists but was
not wired in for this run.

## Part 3: What Worked Because of Prior Planning

### 3.1 register_buffer for lora_scale (zero recompilation on phase transition)

`lora_scale` is a `register_buffer`, not a Python attribute. When
`set_adapter_config("rtheta", scale=0.0)` freezes rtheta for Phase 2, dynamo
guards do not fire — buffer values are not part of the guard condition. Phase
transitions are graph-invariant. This decision was made during initial LoRA
implementation specifically to avoid compile invalidation. Without it, every
`set_adapter_config` call would trigger a full recompile.

**Wallclock saved**: ~14 minutes per phase transition × 2 transitions = ~28 min.
(Based on the 14+ minute recompile observed when graph *was* invalidated by
`inject_lora`.)

### 3.2 custom_op with register_autograd (training works on first forward)

All 12 custom Triton ops have `register_autograd` implementations. The multi-LoRA
sparse kernel, FP8 GEMM fusions, SageAttention backward — all have hand-written
backward passes registered at import time. When `accumulate_policy_gradients()`
ran the first checkpointed forward+backward through the model, autograd found
valid backward functions for every custom op. No "NotImplementedError: backward
not implemented for custom_op" at 2 AM on a rented GPU.

**Wallclock saved**: Incalculable — if autograd had failed, the session would have
been blocked entirely. There is no "quick fix" for a missing backward pass on a
fused Triton kernel.

### 3.3 Dataset v2 sealed blobs (no corruption from partial writes)

DatasetV2Writer accumulates tensors in memory and writes sealed (immutable)
safetensors blobs atomically. No WIP file on disk. If the process is killed
mid-write, the blob simply doesn't exist — the index.parquet still points to
the last successfully sealed blob. No O(N^2) write amplification, no half-written
files, no need for fsync ceremonies.

This was designed specifically for the spot instance use case: preemption can
happen at any time, and the dataset must be valid after ungraceful shutdown.
It wasn't tested under preemption during this session (training completed
normally), but the architecture was ready.

### 3.4 ZMQ JSON envelope protocol (cross-architecture portability)

The client-server protocol uses JSON envelopes with raw tensor bytes (bfloat16
via uint16 view). No pickle. No Python-version-dependent serialization. The
same client code that runs on the local RTX 4090 (SM89, Windows Python,
WSL2 bridge) connected to the H100 server (SM90, Linux native) without any
protocol adaptation. Tensor shapes, dtypes, and RPC schemas are validated at
both ends.

**Wallclock saved**: Unknown, but protocol mismatches are typically diagnosed
by mysterious deserialization errors that don't point at the actual problem.
The explicit JSON schema makes errors immediately legible.

### 3.5 Structured JSONL metrics (complete audit trail)

Every training step writes a JSONL line with wall_clock_s, timestamp, per-head
accuracy, rewards, advantages, grad norms, VRAM usage. The metrics file is the
authoritative record of what happened. When diagnosing the recompile spikes, the
per-iteration rollout_time_s field immediately showed the pattern (51s, 11s, 69s,
28s, 7.9s, 36s, 7.9s...). When checking training health, mean_reward and
grad_norm trends are one `jq` command away.

**Wallclock saved**: ~10 minutes of "is it working? what's happening? let me
SSH in and read the tmux pane" replaced by structured queries against the
metrics file that the rsync loop pulls every 60 seconds.

### 3.6 Health check renders (visual pathology detection)

Every 8 policy iterations, the first rollout's final latent is VAE-decoded,
saved as a PNG, and analyzed for pathology (all-black, all-white, flat, NaN).
Entropy and spectral slope are logged. During this run, all 6 policy render
checks passed (entropy 2.3-7.3, slope -1.8 to -2.8 — healthy natural images).
The BTRM phase produced 3 rtheta renders with similar health.

These renders caught nothing pathological — which is the desired outcome. But
if gradient explosion had produced NaN latents or mode collapse had produced
flat outputs, the render check would have flagged it within 4 minutes (8
iterations × 28s/iter), rather than discovering it after 50 iterations when
examining the final checkpoint.

### 3.7 model_manager.py owns VRAM lifecycle (clean server restarts)

When a server restart was needed (4 times during this session), the restart
cycle was: kill tmux session, re-launch command, wait 30s for model load.
`model_manager.py` handles all VRAM allocation, FP8 weight loading, LoRA
replay, and torch.compile in a single lifecycle owner. There is no "stale
CUDA context" or "leaked GPU memory" because the process exits cleanly and
the new process starts from a blank VRAM state.

**Wallclock saved**: ~5 minutes per restart that would have been spent
debugging CUDA OOM from leaked allocations or stale compiled graph caches
in a long-lived process with mutable model state.

### 3.8 The defects document itself

`docs/live_run_defects_20260216.md` was started during run 1 and updated
after each defect discovery. By run 4, it contained 21 numbered defects
with root cause analysis and fix descriptions. When a new manifestation
of an old defect appeared (e.g., ZMQ EFSM in `generate_trajectories`
was a new manifestation of defect #20's root cause), the existing analysis
provided immediate context: "this is the same socket-safety problem, but
in a different code path."

The document also served as a TODO list for between-run fixes. The
sequence was: discover defect → document it → fix if possible mid-session
→ if not, document the fix for next session → restart → repeat. Each
restart was faster because fewer defects remained.

## Part 4: Provision Hygiene

### The rsync carpet-bomb

`remote.py provision` rsynced the entire repository to the remote server,
including:
- `bench_renders/` (79MB of local SM89 renders)
- `btrm_data/` (48MB of local trajectory data)
- `heattest_renders/` (68KB of local analysis)
- `validation_renders/` (200KB of local residual analysis)
- `stream_comfyui/`, `stream_compat_bf16/`, `stream_futudiffu_bf16/`,
  `stream_futudiffu_f16te/` (multiple MB of reference trajectory tensors)

None of these were needed on the remote. The remote needed: `src/`, `scripts/`,
`pyproject.toml`, `.supersekrit`, and the dataset dirs. The carpet-bomb
provisioned ~400MB of unnecessary data, obscured the provenance of files on
the remote (are these SM90-generated or SM89-rsynced?), and created the
double-named staging dir bug (`btrm_dataset_v2_gpu0_gpu0`) when the remote
generation script appended `_gpu0` to a directory that already ended in `_gpu0`
from the local naming convention.

**Fix needed**: `RSYNC_PUSH_INCLUDE` whitelist instead of (or in addition to)
`RSYNC_PUSH_EXCLUDE` blacklist. Provision should transfer code + config + auth,
not data artifacts. Data generation happens on the remote; data flows back
via rsync pull + HF upload, not forward via provision.

## Part 6: What This Means for the Next Session

### The 8 remaining defects (not fixed live)

1. No `update_btrm_head` RPC (#2) — blocks loading trained BTRM across GPUs
2. No auto-discovery of server count (#4) — easy to forget a GPU
3. No `remote.py generate` / `remote.py train` (#7) — manual SSH + tmux
4. No server readiness signal (#12) — sleep-loop polling
5. No `--model-dir` auto-discovery (#13) — typing wrong filenames
6. No `remote.py upload` subcommand (#23) — manual tmux launch
7. `inject_lora` not formally deprecated (#18) — convenience wrapper exists
8. Provision rsync whitelist — carpet-bomb hygiene

### What the next run should be able to do in <5 minutes setup

1. `remote.py provision` (selective, not carpet-bomb)
2. `remote.py bootstrap` (uv sync)
3. `remote.py models --hf` (if needed)
4. `remote.py launch` (servers with readiness wait)
5. `remote.py upload` (HF tee, automatic)
6. `remote.py train --resume --btrm-checkpoint training_output/final`

Steps 1-5 should be automatable into `remote.py deploy`. Step 6 is the only
one that requires a human decision (which checkpoint, which hyperparameters).

### The trained adapters are portable

The final checkpoint (`training_output/final/`) contains:
- `ptheta_*.safetensors`: 204 tensors, 10.1M params (policy, all layers)
- `rtheta_*.safetensors`: 12 tensors, 594K params (reward LoRA, layers 28-29)
- `btrm_head_*.safetensors`: 2 tensors, 11.5K params (reward model head)

These are architecture-portable (safetensors, explicit shapes, bfloat16).
They can be loaded on the local RTX 4090, a different H100 instance, or any
future machine with the same model architecture. The dump manifest records
rank, alpha, scale, and head config — everything needed to reconstruct the
adapter injection sequence.

The policy can be evaluated locally. The BTRM head can score new trajectories
locally. The reward LoRA can be used as initialization for the next BTRM
training iteration. None of this requires the original H100 instance.
