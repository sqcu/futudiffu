# From DDPO to DDGRPO: Policy Gradient Methods for Diffusion Models

This document derives a family of policy gradient algorithms for fine-tuning diffusion
models with reward signals. It starts from DDPO (Black et al., 2023), strips it to
DDREINFORCE, contrasts with reward-weighted likelihood (RWL), and then constructs
DDGRPO -- the application of DeepSeek's GRPO (Group Relative Policy Optimization)
to the denoising diffusion setting. Section 5 covers implementation considerations
specific to this codebase (FP8 flow-matching inference, sparse step sampling, BTRM
reward model).

Notation conventions: bold lowercase for tensors/images (x), plain lowercase for
scalars (t, sigma), Greek for policies/parameters (theta, pi). Subscript t is the
denoising step index (T = noisiest, 0 = cleanest). Subscript k indexes trajectories
within a group.

---

## 1. DDPO: Denoising Diffusion Policy Optimization

**Source**: Black, Janner, Du, Kostrikov, Levine. "Training Diffusion Models with
Reinforcement Learning." NeurIPS 2023. arXiv:2305.13301.

### 1.1 The Denoising MDP

DDPO's key insight is that a T-step denoising process is a T-step Markov Decision
Process. The components are:

| MDP element | Definition |
|---|---|
| State | s_t = (c, t, x_t) -- context/prompt c, timestep index t, noisy image x_t |
| Action | a_t = x_{t-1} -- the denoised image at the previous noise level |
| Policy | pi_theta(a_t \| s_t) = p_theta(x_{t-1} \| x_t, c) |
| Transition | P(s_{t+1} \| s_t, a_t) = (delta_c, delta_{t-1}, delta_{x_{t-1}}) -- deterministic |
| Initial state | rho_0 = (p(c), delta_T, N(0, I)) |
| Reward | R(s_t, a_t) = r(x_0, c) if t=0, else 0 |

The reward is terminal: only the final denoised image x_0 receives a scalar reward
r(x_0, c) from an external reward function. All intermediate rewards are zero.
The cumulative return of any trajectory therefore equals r(x_0, c).

### 1.2 The Policy as a Gaussian

For a DDPM-style model, the reverse process at each step is an isotropic Gaussian:

```
p_theta(x_{t-1} | x_t, c) = N(x_{t-1} ; mu_theta(x_t, c, t), sigma_t^2 I)
```

where sigma_t is a fixed, timestep-dependent variance (not learned) and mu_theta
is the mean predicted by the neural network. This is critical: the log-probability
of a single denoising step has a closed-form expression:

```
log p_theta(x_{t-1} | x_t, c) = -||x_{t-1} - mu_theta(x_t, c, t)||^2 / (2 sigma_t^2)
                                  - (d/2) log(2 pi sigma_t^2)
```

where d is the dimensionality of x. The second term is constant w.r.t. theta, so
for gradient computation:

```
nabla_theta log p_theta(x_{t-1} | x_t, c) = (x_{t-1} - mu_theta(x_t, c, t)) / sigma_t^2
                                               * nabla_theta mu_theta(x_t, c, t)
```

This is the per-step score function. It is exact (no approximation), cheap to compute
(one forward pass to get mu_theta, then a squared-error against the sampled x_{t-1}),
and differentiable through the network.

### 1.3 The RL Objective

The objective is to maximize expected reward over denoising trajectories:

```
J_DDRL(theta) = E_{c ~ p(c), x_0 ~ p_theta(x_0|c)} [r(x_0, c)]
```

This is the standard RL return objective. The question is how to estimate its gradient.

### 1.4 DDPO-SF (Score Function Estimator / REINFORCE)

Apply the likelihood ratio trick to each step of the trajectory:

```
nabla_theta J_DDRL = E [ sum_{t=0}^{T} nabla_theta log p_theta(x_{t-1} | x_t, c) * r(x_0, c) ]
```

This is REINFORCE applied per-step. The reward r(x_0, c) is the same for every step
in a trajectory (since it is the terminal reward), but the score function
nabla_theta log p_theta differs at each step because each step conditions on a
different (x_t, t).

**Key property**: The expectation must be taken over trajectories sampled from the
current policy p_theta. This makes DDPO-SF on-policy: each gradient update requires
fresh trajectories, and old trajectories cannot be reused.

### 1.5 DDPO-IS (Importance Sampling / PPO)

To reuse trajectories generated by a previous policy pi_theta_old, apply importance
sampling:

```
nabla_theta J_DDRL = E_{tau ~ pi_{theta_old}} [
    sum_{t=0}^{T} (p_theta(x_{t-1} | x_t, c) / p_{theta_old}(x_{t-1} | x_t, c))
    * nabla_theta log p_theta(x_{t-1} | x_t, c)
    * r(x_0, c)
]
```

The importance weight at each step is the ratio of Gaussian densities:

```
rho_t = p_theta(x_{t-1} | x_t, c) / p_{theta_old}(x_{t-1} | x_t, c)
      = exp( -||x_{t-1} - mu_theta||^2 / (2 sigma_t^2)
             + ||x_{t-1} - mu_{theta_old}||^2 / (2 sigma_t^2) )
```

(The sigma_t terms cancel because the variance is fixed and does not depend on theta.)

Equivalently, taking the log:

```
log rho_t = log p_theta(x_{t-1} | x_t, c) - log p_{theta_old}(x_{t-1} | x_t, c)
```

**PPO-style clipping** bounds the ratio to prevent destructive updates when the
policy has drifted far from theta_old:

```
L_t^CLIP = min(rho_t * A_t,  clip(rho_t, 1-epsilon, 1+epsilon) * A_t)
```

where A_t is the advantage (in the simplest case, A_t = r(x_0, c) - b, with b a
baseline). The overall DDPO-IS loss (to be maximized, or equivalently, the negative
is minimized) is:

```
L_DDPO-IS = E_{tau ~ pi_{theta_old}} [ sum_{t=0}^{T} L_t^CLIP ]
```

This enables multiple optimization epochs per batch of trajectories, exactly as in
standard PPO. In practice, Black et al. report clip_range = 1e-4 (very conservative),
reflecting the high dimensionality of the action space (a full image at each step).

### 1.6 Per-Step Credit Assignment

A subtlety: the terminal reward r(x_0, c) is applied to EVERY step's gradient.
This is correct under the policy gradient theorem (the gradient of the trajectory
log-probability decomposes as a sum of per-step log-probabilities), but it means
that step T (pure noise) gets the same reward signal as step 0 (final refinement).

The per-step baseline b_t can reduce variance without biasing the estimator.
DDPO uses per-prompt advantage normalization: for each prompt, compute
advantages = (rewards - mean(rewards)) / std(rewards) across the trajectories
generated for that prompt. This is a trajectory-level baseline, not a per-step one.

## 2. Reward-Weighted Likelihood (RWL)

RWL (also called Reward-Weighted Regression, RWR) is the main baseline DDPO compares
against. It treats the entire generation process as a single-step bandit problem.

### 2.1 Formulation

Generate trajectories, compute rewards, then do weighted maximum-likelihood training
on the generated samples:

```
L_RWL(theta) = E_{(x_0, c) ~ D} [ w(x_0, c) * L_DDPM(theta; x_0, c) ]
```

where L_DDPM is the standard diffusion training loss (noise prediction MSE) and
w(x_0, c) is a reward-derived weight. Two common choices:

**Exponential (RWR)**:
```
w_RWR(x_0, c) = (1/Z) * exp(beta * r(x_0, c))
```

**Sparse (RWR-sparse)**:
```
w_sparse(x_0, c) = 1[r(x_0, c) >= C]
```

where C is typically a percentile threshold (e.g., top 25%).

### 2.2 Why RWL is Weaker

RWL has two levels of approximation that DDPO avoids:

1. **Approximate likelihood**: The DDPM loss L_DDPM is an ELBO on log p(x_0), not
   the exact log-likelihood. This means the gradient of the weighted DDPM loss is
   NOT the gradient of the weighted log-probability. DDPO uses the exact per-step
   Gaussian log-probability, which IS exact.

2. **No per-step credit assignment**: RWL treats the entire trajectory as a single
   action. The weight w applies uniformly to all timestep losses within a sample.
   There is no mechanism for the reward signal to be stronger at the steps that
   mattered most (e.g., the final refinement steps vs. early noise removal).

3. **No off-policy correction**: RWL reweights samples but does not account for the
   distributional shift between the policy that generated the samples and the
   current policy being optimized. After one gradient step, the samples are stale.
   DDPO-IS corrects for this via importance sampling.

### 2.3 Summary Table

| Property | RWL | DDPO-SF | DDPO-IS |
|---|---|---|---|
| Likelihood | Approximate (ELBO) | Exact (per-step Gaussian) | Exact (per-step Gaussian) |
| Credit assignment | Trajectory-level | Per-step | Per-step |
| Off-policy correction | None | N/A (on-policy) | Importance sampling + clip |
| Epochs per batch | 1 (but can iterate) | 1 | Multiple (PPO-style) |
| Variance | Low (supervised) | High (REINFORCE) | Medium (IS + clip) |
| Sample efficiency | Low | Low | Higher |

## 3. DDREINFORCE: The Simplest Diffusion Policy Gradient

DDREINFORCE is DDPO-SF stripped to its bare essentials. No importance sampling, no
PPO clipping, no multi-epoch optimization. Pure on-policy REINFORCE with a
per-trajectory baseline.

### 3.1 The Gradient

```
nabla_theta J = E_{c ~ p(c), tau ~ pi_theta(.|c)} [
    (r(x_0, c) - b(c)) * sum_{t=0}^{T} nabla_theta log p_theta(x_{t-1} | x_t, c)
]
```

where b(c) is a baseline (e.g., running mean of rewards for prompt c).

### 3.2 With Per-Step Baselines

A more refined variant subtracts a per-step baseline:

```
nabla_theta J = E [ sum_{t=0}^{T} (r(x_0, c) - b_t(c)) * nabla_theta log p_theta(x_{t-1} | x_t, c) ]
```

where b_t(c) is the expected future return from state s_t. Since all intermediate
rewards are zero, b_t(c) = E[r(x_0, c) | s_t] for all t. In practice this is
intractable without a learned value function, so the trajectory-level baseline
b(c) = E[r(x_0, c) | c] is used instead.

### 3.3 Properties

- **Simplest possible**: No ratio computation, no clipping, no stored log-probs.
- **On-policy only**: Every gradient estimate requires fresh trajectories.
- **High variance**: REINFORCE with a trajectory-level baseline has notoriously
  high variance, especially with T=30 or T=50 denoising steps.
- **Correct**: The estimator is unbiased. Given enough samples, it converges.

### 3.4 Relationship to DDPO

DDREINFORCE = DDPO-SF with no bells and whistles. DDPO-SF as described in the
paper is essentially DDREINFORCE with per-prompt advantage normalization.

## 4. DDGRPO: Group Relative Policy Optimization for Diffusion

**This section is our extension.** It applies GRPO (Shao et al., "DeepSeekMath:
Pushing the Limits of Mathematical Reasoning in Open Language Models," 2024,
arXiv:2402.03300) to the diffusion denoising MDP.

### 4.1 GRPO Review (LLM Setting)

In the LLM setting, GRPO replaces PPO's learned value function (critic) with a
group-relative advantage computation. For each prompt q, generate G completions
{o_1, ..., o_G} from the current policy pi_{theta_old}. Score each with a reward
function to get {r_1, ..., r_G}. The advantage of completion i is:

```
A_hat_i = (r_i - mean(r_1, ..., r_G)) / std(r_1, ..., r_G)
```

The objective is:

```
J_GRPO(theta) = E_{q ~ P(Q), {o_i} ~ pi_{theta_old}(.|q)} [
    (1/G) sum_{i=1}^{G} (1/|o_i|) sum_{t=1}^{|o_i|} {
        min( rho_{i,t} * A_hat_i,  clip(rho_{i,t}, 1-eps, 1+eps) * A_hat_i )
        - beta * D_KL(pi_theta || pi_ref)
    }
]
```

where:
- rho_{i,t} = pi_theta(o_{i,t} | q, o_{i,<t}) / pi_{theta_old}(o_{i,t} | q, o_{i,<t})
  is the per-token importance sampling ratio
- D_KL uses the unbiased estimator: D_KL(pi_theta || pi_ref) = pi_ref/pi_theta
  - log(pi_ref/pi_theta) - 1, which is guaranteed non-negative
- beta controls KL penalty strength (DeepSeek uses beta=0.04)
- pi_ref is the reference policy (typically the initial SFT model)

**Key GRPO property**: No critic network. The baseline is the group mean. This
eliminates the need to train a separate value model, reducing memory by ~50%
compared to PPO.

### 4.2 DDGRPO: Mapping GRPO to the Denoising MDP

The translation is direct. Replace LLM tokens with denoising steps, replace the
autoregressive policy with the Gaussian denoising policy.

**Group generation**: For each prompt c, generate K denoising trajectories
{tau_1, ..., tau_K} where tau_k = {x_T^k, x_{T-1}^k, ..., x_0^k} from the
current policy pi_{theta_old}. Each trajectory produces a final image x_0^k.

**Reward and advantage**: Score each final image:

```
r_k = r(x_0^k, c)
```

Compute the group-relative advantage:

```
A_hat_k = (r_k - mean(r_1, ..., r_K)) / (std(r_1, ..., r_K) + epsilon_std)
```

where epsilon_std (e.g., 1e-8) prevents division by zero when all trajectories
receive the same reward.

**Per-step importance ratio**: For each trajectory k and denoising step t:

```
rho_{k,t} = p_theta(x_{t-1}^k | x_t^k, c) / p_{theta_old}(x_{t-1}^k | x_t^k, c)
```

Since both densities are Gaussian with the same variance sigma_t^2:

```
log rho_{k,t} = (1 / (2 sigma_t^2)) * (
    ||x_{t-1}^k - mu_{theta_old}(x_t^k, c, t)||^2
  - ||x_{t-1}^k - mu_theta(x_t^k, c, t)||^2
)
```

**The DDGRPO objective**:

```
J_DDGRPO(theta) = E_{c ~ p(c)} [
    (1/K) sum_{k=1}^{K} (1/T) sum_{t=0}^{T} {
        min( rho_{k,t} * A_hat_k,  clip(rho_{k,t}, 1-eps, 1+eps) * A_hat_k )
        - beta * D_KL^t(pi_theta || pi_ref)
    }
]
```

where:

```
D_KL^t(pi_theta || pi_ref) = p_{theta_ref}(x_{t-1}^k | x_t^k, c) / p_theta(x_{t-1}^k | x_t^k, c)
                             - log( p_{theta_ref}(x_{t-1}^k | x_t^k, c) / p_theta(x_{t-1}^k | x_t^k, c) )
                             - 1
```

is the per-step unbiased KL estimator from DeepSeek-V3, guaranteed non-negative.

### 4.3 Why DDGRPO Over DDPO-IS

| Property | DDPO-IS | DDGRPO |
|---|---|---|
| Advantage computation | Per-prompt mean/std of rewards | Same, but explicitly group-normalized |
| Critic network | Not used (per-prompt stats tracker) | Not used (group normalization) |
| KL regularization | Implicit (clip only) | Explicit beta * D_KL term |
| Reference policy | theta_old only | Separate pi_ref (frozen initial model) |
| Advantage scope | All trajectories in a batch | Within-prompt group only |

The distinction between DDPO-IS with per-prompt normalization and DDGRPO is subtle
but important:

1. **Explicit KL penalty**: DDGRPO penalizes divergence from a fixed reference policy
   pi_ref, not just from theta_old. This prevents reward hacking / mode collapse
   over extended training, which DDPO-IS observed empirically (they call it
   "overoptimization").

2. **Formal group structure**: GRPO mandates that advantages are computed WITHIN a
   group of completions for the same prompt. DDPO's per-prompt stat tracker is an
   ad-hoc approximation of this (it uses a running exponential average, which mixes
   statistics across optimization steps).

3. **Composability with process supervision**: GRPO's framework supports per-step
   advantages when step-level rewards are available. In the diffusion setting, if a
   reward model can score intermediate noise levels (as our BTRM can), the advantage
   can be refined:

```
A_hat_{k,t} = (r_{k,t} - mean_j(r_{j,t})) / (std_j(r_{j,t}) + epsilon_std)
```

where r_{k,t} is the reward at step t of trajectory k. This is process supervision
applied to diffusion -- a strictly stronger signal than terminal-only reward.

### 4.4 Simplified DDGRPO (No Clipping, On-Policy)

For an on-policy implementation where fresh trajectories are generated every step
(no trajectory reuse), the importance ratios are identically 1 and the clip is
vacuous. The objective simplifies to:

```
J_DDGRPO-simple(theta) = E_{c ~ p(c)} [
    (1/K) sum_{k=1}^{K} A_hat_k * sum_{t=0}^{T} log p_theta(x_{t-1}^k | x_t^k, c)
    - beta * D_KL(pi_theta || pi_ref)
]
```

This is DDREINFORCE (Section 3) with the group-relative advantage replacing the
simple baseline, plus a KL penalty. It is the minimum viable DDGRPO.

### 4.5 The Complete DDGRPO Algorithm

```
Algorithm: DDGRPO (Denoising Diffusion Group Relative Policy Optimization)
------------------------------------------------------------------------------
Input: pi_theta (current policy), pi_ref (frozen reference policy),
       r(.) (reward function), K (group size), T (denoising steps),
       eps (clip range), beta (KL coefficient), N_epochs (PPO epochs per batch)

For each training iteration:
  1. Sample a batch of prompts {c_1, ..., c_B}
  2. For each prompt c_b:
     a. Generate K trajectories: tau_k = sample(pi_theta, c_b, T)  for k=1..K
     b. Record per-step log-probs: log_pi_old[k,t] = log p_theta(x_{t-1}^k | x_t^k, c_b)
     c. Compute rewards: r_k = r(x_0^k, c_b)
     d. Compute group advantages: A_hat_k = (r_k - mean(r)) / (std(r) + eps_std)
  3. For epoch = 1 to N_epochs:
     For each prompt c_b and trajectory k:
       a. Recompute log-probs: log_pi_new[k,t] = log p_theta(x_{t-1}^k | x_t^k, c_b)
       b. Compute ratios: rho[k,t] = exp(log_pi_new[k,t] - log_pi_old[k,t])
       c. Compute KL: kl[k,t] = exp(log_pi_ref[k,t] - log_pi_new[k,t])
                                 - (log_pi_ref[k,t] - log_pi_new[k,t]) - 1
       d. Compute per-step loss:
          L[k,t] = -min(rho[k,t] * A_hat_k, clip(rho[k,t], 1-eps, 1+eps) * A_hat_k)
                   + beta * kl[k,t]
       e. Aggregate: L_total = mean over (b, k, t) of L[k,t]
       f. Backprop and optimizer step.
```

## 5. Implementation Considerations for FP8 Diffusion with Sparse Step Sampling

This codebase has several properties that affect how DDGRPO would be implemented.

### 5.1 Flow Matching, Not DDPM

This model uses a CONST noise model (flow matching / rectified flow), not a DDPM
Gaussian noise model. The forward process is:

```
x_t = sigma * noise + (1 - sigma) * x_0
```

where sigma in [0, 1] (sigma=1 is pure noise, sigma=0 is clean image). The reverse
process is an ODE solved by Euler steps:

```
x_{t-1} = x_t + (sigma_{t-1} - sigma_t) * v_theta(x_t, c, t)
```

where v_theta is the learned velocity field.

**Problem**: The Euler ODE step is deterministic. There is no stochasticity and
therefore no Gaussian log-probability to compute. The entire DDPO framework
assumes p_theta(x_{t-1} | x_t, c) = N(mu_theta, sigma_t^2 I), which does not
apply to deterministic ODE solvers.

**Solutions**:

(a) **Inject stochasticity**: Add noise at each step to create an SDE:
```
x_{t-1} = x_t + (sigma_{t-1} - sigma_t) * v_theta(x_t, c, t) + eta_t * z
```
where z ~ N(0, I) and eta_t is a noise scale. Then:
```
log p_theta(x_{t-1} | x_t, c) = -||x_{t-1} - mu_theta(x_t, c, t)||^2 / (2 eta_t^2) + const
```
This is the standard approach. The noise eta_t is a hyperparameter that trades off
exploration (more noise) vs. sample quality (less noise). DDPO's original codebase
uses DDIM with non-zero eta for exactly this reason.

(b) **FPO-style ratio estimation** (Akhound-Sadegh et al., 2025, arXiv:2507.21053):
Instead of computing exact log-probabilities, use the flow matching loss as a
surrogate for the ELBO, and estimate the importance ratio as:
```
rho^FPO = exp(L_{theta_old}(x) - L_theta(x))
```
where L_theta(x) = ||v_theta(x^tau, tau; c) - (x - epsilon)||^2 is the per-sample
conditional flow matching loss. This avoids injecting stochasticity entirely but
requires multiple Monte Carlo draws of (tau, epsilon) per sample to reduce variance.

(c) **Reward-only optimization**: If we only need the gradient of the reward w.r.t.
theta (not the log-probability), and the reward model is differentiable, we can
backprop through the entire denoising chain. This is what BTRM training currently
does (score_differentiable_packed). This sidesteps the policy gradient formulation
entirely but requires a differentiable reward function and cannot handle black-box
rewards.

**Recommendation for this codebase**: Approach (a) is simplest and most compatible
with DDGRPO. The CONST noise model's Euler step can be augmented with a small
eta_t at each step to produce a stochastic policy. eta_t should be proportional
to |sigma_{t-1} - sigma_t| (the step size) to keep the noise scale commensurate
with the signal change.

### 5.2 Sparse Step Sampling

The dataset generation pipeline saves 7 intermediate steps per 30-step trajectory
(selected by logSNR-uniform spacing, not step-index-uniform), plus the final
denoised image (sigma=0). The full 30-step trajectory is NOT stored.

For DDGRPO, this means:

1. **Generating trajectories**: Full 30-step trajectories must be generated on-the-fly
   during training (not loaded from the dataset). The dataset is for BTRM (reward
   model) training, not policy gradient training.

2. **Sparse reward signals**: The BTRM can score images at any saved step, providing
   per-step rewards r(x_t, c) at the 7+1 saved positions. This enables process
   supervision (Section 4.3) at these positions, with zero reward signal at the
   ~23 unsaved positions.

3. **logSNR-weighted step sampling**: Rather than summing the policy gradient over
   all T steps with equal weight, weight each step's contribution by its logSNR
   sampling weight (the existing geometric decay schedule). This focuses the
   gradient on the clean-regime steps where the BTRM signal is strongest:
   ```
   nabla J approx sum_{t in saved_steps} w_logSNR(sigma_t) * nabla_theta log p_theta * A_hat
   ```

4. **Sigma=0 dominance**: The cleanest step (sigma=0, fully denoised) gets full
   weight=1.0 and carries the most informative reward signal. Under the current
   clean_fraction=0.8 sampling bias, 80% of training signal comes from sigma=0
   positions. This is desirable: the reward model is most accurate on clean images,
   and the policy gradient at sigma=0 directly optimizes the final output quality.

### 5.3 FP8 Precision and Gradient Flow

The inference backbone runs FP8 weights with BF16 I/O. For policy gradient training:

- **Forward pass**: FP8 matmuls with FP32 accumulation (existing infrastructure).
  The 12 custom_ops with register_autograd support training backward.
- **Log-probability computation**: The per-step log-prob requires the predicted
  mean mu_theta (a BF16 tensor from the forward pass) and the sampled x_{t-1}
  (also BF16). The squared difference and division by sigma_t^2 should be computed
  in FP32 to avoid underflow (sigma_t^2 can be very small near sigma=0).
- **Importance ratios**: exp(log_pi_new - log_pi_old) is numerically sensitive.
  Compute in FP32. The clip bounds this to [1-eps, 1+eps], limiting damage from
  numerical noise.
- **KL penalty**: The unbiased estimator exp(x) - x - 1 (where x = log(pi_ref/pi_theta))
  is well-conditioned near x=0 (where pi_theta ~ pi_ref), but requires FP32 for
  the exp/log operations.

### 5.4 BTRM as Reward Function

The BTRM (Bradley-Terry Reward Model) produces multi-headed scores:
- Head 0 ("pinkify"): discriminates quantization artifacts
- Head 1 ("thisnotthat"): discriminates step count

These are scalar scores per image, suitable as reward signals for DDGRPO:

```
r_k = alpha_pinkify * score_pinkify(x_0^k) + alpha_tnt * score_tnt(x_0^k)
```

or as separate reward signals for multi-objective optimization (one DDGRPO run per
head, or multi-head advantages).

**Differentiable path**: Because the BTRM is differentiable (it is a LoRA adapter
on the frozen backbone), we can also compute nabla_theta r(x_0, c) directly. This
gives us the option of combining policy gradients (for exploration and off-policy
correction) with direct reward gradients (for local refinement):

```
nabla_theta J_hybrid = nabla_theta J_DDGRPO + lambda * E[nabla_theta r(x_0, c)]
```

The second term is the "reward gradient" -- it pushes the policy toward locally
increasing the reward, without the variance of the policy gradient. This is related
to DRGPO (Direct Reward Gradient Policy Optimization), which is noted in CLAUDE.md
as a future direction.

### 5.5 Group Size and VRAM Budget

Generating K trajectories per prompt requires K full forward passes through the
6B parameter backbone (each: ~8GB VRAM with torch.compile). With 30 Euler steps
per trajectory:

- K=4, T=30: 120 forward passes per prompt per training step
- K=8, T=30: 240 forward passes per prompt per training step

This is expensive. Mitigations:

1. **Sparse step evaluation**: Only compute log-probs at the 7+1 saved step positions,
   not all 30. The unsaved steps contribute zero gradient signal anyway.

2. **Sequential generation, parallel scoring**: Generate trajectories one at a time
   (sequential in VRAM), but score all K final images in a packed batch using
   FlexAttention batch packing (existing infrastructure).

3. **Small K with cross-prompt grouping**: If K=2 per prompt but B=16 prompts per
   batch, the effective group for advantage normalization can be expanded by
   normalizing across all B*K trajectories rather than within-prompt groups. This
   loses the "same-prompt" property of GRPO but reduces generation cost.

4. **Checkpoint-and-recompute**: Use activation checkpointing (forward_checkpointed)
   for the backward pass, trading 2x compute for ~60% VRAM reduction.

### 5.6 Per-Step Sigma Schedule and KL Scaling

The sigma schedule is resolution-dependent (SD3 Eq.23 shift). For a given resolution,
the step sizes dt = sigma_{t-1} - sigma_t vary -- they are larger in the high-noise
regime and smaller in the clean regime. This means:

- The injected noise eta_t (for stochastic policy, Section 5.1a) should be
  proportional to |dt| to maintain a consistent signal-to-noise ratio in the
  stochastic perturbation.
- The per-step KL penalty D_KL^t should be weighted by the step size to avoid
  over-penalizing small steps in the clean regime:
  ```
  weighted_kl = |dt| * D_KL^t
  ```
  or equivalently, use the logSNR sampling weight as the KL weight.

### 5.7 Compatibility with Existing Training Infrastructure

The DDGRPO algorithm fits naturally on top of the existing codebase:

| DDGRPO component | Existing infrastructure |
|---|---|
| Trajectory generation | server.py / model_manager.py (inference pipeline) |
| Per-step log-prob | New: requires stochastic injection + log-prob tracking |
| Reward scoring | BTRMCompoundModel.score() + score_differentiable_packed() |
| Group advantage | New: simple mean/std normalization per prompt group |
| PPO clipping | New: standard clip implementation |
| KL penalty | New: unbiased estimator against frozen reference |
| Step weighting | Existing: logsnr_sampling_weight from pair_sampler.py |
| Batch packing | Existing: FlexAttention + bin packer for multi-image scoring |
| Sigma schedule | Existing: sigma_schedule.py with resolution shift |
| Activation checkpointing | Existing: forward_checkpointed in training_utils.py |

The main new components are: (1) stochastic policy injection during trajectory
generation, (2) per-step log-probability tracking, (3) the DDGRPO loss function
with group advantages and KL penalty. Everything else is a composition of
existing modules.
