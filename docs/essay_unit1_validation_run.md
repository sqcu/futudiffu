# Essay: Unit 1 Validation Run — src_ii Rollout vs Reference Trajectory

## What Broke

When the task was assigned, the script `scripts_ii/validate_trajectory.py` had already been through one round of debugging: WSL-format paths had been replaced with Windows-format paths (using `REPO_ROOT` derived from `__file__`), and the `sys.path` insertion had been corrected to point at `REPO_ROOT` rather than `REPO_ROOT / "src_ii"`. The task description listed four known-fixed issues and flagged several likely remaining issues — path problems in `src_ii/model_loading.py`, WSL path leakage into module internals, and potential API mismatches between `src_ii/rollout.py` and the rest of the stack. In practice, none of these remaining issues materialized. The script ran without modification on the first attempt.

## What Was Fixed

Nothing required fixing in this session. The prior debugging passes had already resolved the structural issues. `src_ii/model_loading.py` correctly uses `futudiffu.diffusion_model` and `futudiffu.fp8` via package imports that work through `sys.path`, and all paths passed to those modules are Windows-format strings compatible with the Windows Python interpreter. The `src_ii/rollout.py` module calls `futudiffu.attention.set_attention_backend` and `futudiffu.diffusion_model` indirectly through the model reference, not through path operations, so there was no WSL/Windows conversion issue. The five-function decomposition (`sigma_schedule`, `forward`, `guided_denoiser`, `solver`, `rollout`) is self-consistent and required no patching. The run completed all four phases — prompt encoding, model loading, rollout, comparison — in approximately 51 seconds of wall clock time (plus CUDA warm-up overhead for the text encoder compilation).

## What the Comparison Stats Show

The comparison against trajectory 0 (t2i, SDPA precision, 30 steps, seed 478163327, 1280x832, "Bauhaus-inspired geometric composition...") produced the following relative L2 norms at each saved step: step_00 = 0.000 (exact match), step_04 = 0.024, step_09 = 0.101, step_14 = 0.187, step_19 = 0.250, step_24 = 0.288, step_29 = 0.315, final = 0.319. The initial noise (step_00) is a perfect zero-diff, confirming that the seeded `torch.Generator` produces identical noise. The divergence then accumulates monotonically across steps, reaching about 32% relative L2 at the end. This pattern is consistent with floating-point non-determinism compounding over 30 Euler steps — the two trajectories used different execution environments (original server run vs. this direct load), and potentially different torch versions, compilation states, or CUDA op orderings that produce identical results on step_00 but diverge by step_04. The max absolute difference at the final step is 5.70 in BF16 latent space, which is large in absolute terms but normal for a 30-step chaotic integration.

## Why It Could Not Be Exact

The reference trajectories in `btrm_dataset/` were generated by the inference server running through the ZeroMQ RPC layer with `torch.compile` active, likely under a specific fused kernel configuration. This validation run used `--no-compile` (eager mode) and loaded the model directly rather than through `ModelManager`. Even with identical seeds and parameters, `torch.compile` versus eager mode can produce subtly different floating-point orderings due to kernel fusion changing the sequence of rounding events. Additionally, the server path applies `configure_sage_if_needed` before the model's weights are finalized on device, while this standalone loader applies `set_attention_backend` after loading. None of these differences produce logically wrong outputs — they produce numerically distinct-but-equivalent outputs. To achieve bit-exact reproduction, the validation would need to match the original compilation mode and kernel selection exactly, which was not the goal of this run.

## What Remains

The primary remaining question is whether the ~32% relative L2 drift is acceptable for downstream tasks or whether it indicates a pipeline configuration mismatch that should be closed. The monotonically increasing pattern rules out a single large divergence event (e.g. a wrong operator) and suggests accumulated floating-point non-determinism, which is expected and benign. The next useful comparison would be to run with `--no-no-compile` (i.e. with `torch.compile` active) to determine how much of the gap closes; if the drift collapses near zero that confirms compilation state as the cause, while if it stays the same it points to a kernel-selection or weight-loading difference. Additionally, the current comparison is single-trajectory; expanding to trajectories 1-4 (same SDPA t2i regime) would confirm whether the drift statistics are stable across seeds, and to trajectories 5-9 (SAGE precision) would require either enabling SageAttention locally or accepting the known backend mismatch.
